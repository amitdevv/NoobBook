| 23 | ⬜ | **Chat list: collapse N+1 message-count queries** | `backend/app/services/data_services/chat_service.list_chats()` (lines 42-75) fetches all chats in 1 query, then loops calling `.select('id', count='exact').eq('chat_id', chat['id'])` per chat for the `message_count` field shown in the sidebar. **20 chats = 21 round-trips on every sidebar render** (every page load + every chat sidebar refresh). On a project with active chat history this is the dominant latency contributor on dashboard load. **Two paths, pick one:** (a) **Aggregate query** — Supabase RPC that runs `SELECT chats.*, (SELECT count(*) FROM messages WHERE messages.chat_id = chats.id) AS message_count FROM chats WHERE project_id = ? ORDER BY updated_at DESC` and returns the joined row. One round-trip total. (b) **Denormalize** — add a `message_count INT NOT NULL DEFAULT 0` column on `chats` via a new migration; increment from `message_service.create_message()` and decrement on delete. Every later list call becomes O(1) at the cost of write-side bookkeeping. Option (a) is non-invasive; option (b) is faster long-term but requires the column + an integrity backfill for existing rows. **Estimated win:** 80-150ms latency on the chat sidebar for projects with 20+ chats. |
