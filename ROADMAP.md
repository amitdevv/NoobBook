| 18 | 🟡 | **Session length / refresh-token TTL + AuthPage refresh** ([screenshot](docs/roadmap/18-session-logout.png)) | Customer (`pmbot.dericrypt.com`) reports being logged out very frequently — well before the access-token's 1-hour TTL would suggest. Diagnosis: access tokens are correctly configured (`JWT_EXPIRY=3600` in `docker/supabase/.env:58`, propagated to `GOTRUE_JWT_EXP` / `PGRST_APP_SETTINGS_JWT_EXP` / DB `JWT_EXP` in `docker/supabase/docker-compose.yml:118,154,375`). Frontend silent refresh is correctly wired (`frontend/src/lib/api/client.ts:62-125` — shared `refreshPromise` dedup + retry on 401 → `POST /auth/refresh`), and the backend endpoint (`backend/app/api/auth/routes.py:91-107`) calls Supabase's native `refresh_session`. **Root cause:** no `GOTRUE_REFRESH_TOKEN_*` / `GOTRUE_SESSION_*` env vars are set anywhere — refresh-token TTL falls back to GoTrue's compiled default, which can be as short as 1 hour in some builds. When the refresh token expires, `tryRefreshToken()` calls `clearSession()` (`client.ts:80`) and the user is logged out without warning. **Fix:** (a) add `GOTRUE_JWT_DEFAULT_GROUP_NAME` is fine, but explicitly set `GOTRUE_REFRESH_TOKEN_REUSE_INTERVAL=10` (10s reuse window for retry safety) and ensure refresh-token rotation is enabled. Most importantly, set the access-token bump to ~24h via `JWT_EXPIRY=86400` so silent-refresh races stop biting on flaky networks. (b) Surface a non-blocking toast when silent refresh fails ("Session expired — re-sign in") instead of silently clearing. (c) Add a `/auth/me` background ping every 5 min while the tab is visible to keep the refresh token warm. Verify by leaving a tab open for 24h and confirming no logout. |
